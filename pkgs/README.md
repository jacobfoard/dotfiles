# Custom Packages

This sub-flake contains custom package definitions and an overlay.

## Structure

Each package is a `.nix` file in the `pkgs/` directory:

```
pkgs/
├── flake.nix              # Exports overlay + direct package outputs
├── argocd-mcp.nix         # ArgoCD MCP server (Node.js)
├── av.nix                 # Aviator stacking tool (Go)
├── beads.nix              # Beads issue tracker (Go)
├── ccstatusline.nix       # Claude Code status line (Bun/Node.js, uses bun2nix)
├── ccstatusline-bun.nix   # Generated bun2nix lockfile (do not edit manually)
├── gastown.nix            # Gastown (Go, currently disabled -- sandbox issue)
└── kubectl-cnpg.nix       # kubectl-cnpg plugin (Go)
```

## Usage

The overlay is exported as `overlays.default` and applied by both the `darwin` and `home-manager` sub-flakes, making all custom packages available everywhere.

Build a specific package:
```bash
nix build ./pkgs#av
nix build ./pkgs#beads
nix build ./pkgs#ccstatusline
```

## Updating Packages

Standard Go packages (av, beads, kubectl-cnpg):
```bash
make update-pkgs
```

ccstatusline (requires extra bun2nix step):
```bash
make update-ccstatusline [VERSION=x.y.z]
```

These Makefile targets use `nix-update` to bump versions/hashes and auto-commit.

## Adding a New Package

1. Create `pkgs/<name>.nix` with the package derivation
2. Add it to the overlay in `flake.nix` (`overlays.default`)
3. Add it to the `packages` output in `flake.nix`
4. Reference it from wherever needed (home-manager packages, darwin machine config, etc.)

## Notes

- The `nixpkgs-staging` input is a temporary workaround for Go version pinning. It will be removed once `nixpkgs-unstable` catches up.
- `ccstatusline-bun.nix` is generated by `bun2nix` and should not be edited manually.
- `gastown` is disabled due to a sandbox build issue (bd version timeout during completion generation).
